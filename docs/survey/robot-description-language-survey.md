# 机器人与机构描述语言调查报告

## 1. 调查目的

本报告面向当前项目“模块化闭环机构拓扑编码 -> 自动生成运动学方程”的阶段 A 需求，调查现有主流机器人与机构描述语言，重点回答以下问题：

1. 它们如何描述刚体、关节、局部坐标系和几何参数。
2. 它们如何处理参数复用、模块组合和文件拆分。
3. 它们是否原生支持闭环、约束、柔性、驱动和传感器。
4. 哪些设计值得借鉴到本项目的自定义符号化语言。

本报告的目标不是为项目选一个现成格式直接使用，而是提炼一套对本项目最有价值的语言设计原则。

---

## 2. 评估维度

为了和本项目目标对齐，以下语言统一按这几个维度比较：

- **表达层级**：它更像几何装配格式、运动学格式、仿真格式，还是带宏的生成语言。
- **模块化能力**：是否支持模板、参数化、复用、文件组合和命名空间。
- **坐标系表达**：是否把 frame / port / 局部参考系作为一等概念。
- **闭环与约束**：是否能直接表达闭环、耦合关系、等式约束或外部约束元件。
- **解释器友好度**：是否容易编写解析器，把文本结构翻译成中间运动学图。
- **对本项目的借鉴价值**：哪些机制值得吸收，哪些机制应该避免照搬。

---

## 3. 总览结论

如果只看与你当前目标最相关的部分，可以先给一个压缩结论：

- **URDF** 适合借鉴“link/joint 树式描述”和基础坐标系写法，但它对闭环和模块化复用支持偏弱，不能直接作为你的目标语言原型。
- **Xacro** 最值得借鉴的是“参数化模板层”，但它本质是文本展开器，不是语义层解释器。
- **SDFormat (SDF)** 在模型组合、frame、嵌套模型、作用域命名上明显比 URDF 更成熟，是非常重要的参考对象。
- **MJCF** 在“编译式建模”和“中间 meta 层”上很强，尤其值得借鉴 `frame`、`include`、`replicate`、`attach`、`equality` 这些机制。
- **Modelica MultiBody** 最值得借鉴的是“组件连接 + 约束驱动 + 闭环天然存在”的思想，它更接近“可组合机构语言”而不是“机器人树描述文件”。
- **Simscape Multibody** 与你当前模块库最接近，但它的优势主要在图形化建模和求解器集成，不适合作为文本 DSL 直接继承对象。
- **Drake Model Directives / Drake Extensions** 值得借鉴“装配层”和“基础格式之上的扩展层”这两个设计思路，尤其适合你未来做模块库拼装语言。
- **DH 参数体系** 不适合作为完整描述语言，但适合作为某些开环链的低级几何后端或验证基线。

因此，如果你要为当前项目自定义一套语言，最合理的借鉴组合不是“选一种现成语言”，而是：

- 用 **URDF/SDF** 的 link-joint-frame 基本骨架。
- 用 **Xacro** 的参数化与模板化思想。
- 用 **MJCF / SDF / Drake** 的组合、命名空间、编译期展开和约束机制。
- 用 **Modelica / Simscape** 的“连接语义优先于文件语法”的理念。

## 4. 各语言调查

## 4.1 URDF

### 定位

URDF 是 ROS 体系中最经典的机器人描述格式，核心是用 XML 描述：

- `link`
- `joint`
- `visual`
- `collision`
- `inertial`

它天然偏向 **树形刚体系统描述**，非常适合串联机器人、移动机器人、传感器挂载等典型结构。

从 `urdfdom` 当前实现可见，URDF 至少稳定覆盖：

- 链接与关节定义
- 常见关节类型：`revolute`、`continuous`、`prismatic`、`fixed`、`floating`、`planar`
- 几何体和网格
- `xyz + rpy` 位姿描述
- 新版还加入了 `quat_xyzw` 和更丰富的 joint limit 语义

### 优点

1. **概念极简清楚。** `link` 和 `joint` 的划分非常稳定，适合作为解释器的基础抽象。
2. **生态极强。** 解析器、可视化工具、转换工具很多，便于后续互操作。
3. **非常适合开环。** 你的阶段 A 从开环先做，URDF 的树结构对于第一版解释器是很好的参照物。
4. **姿态写法直观。** `origin` + `axis` 的写法简单直接，适合作为文本 DSL 的启发。

### 缺点

1. **本质是树，不是一般图。** 对闭环、跨链约束、多环结构表达很弱。
2. **模块化复用能力弱。** 标准 URDF 本身没有宏、模板、实例化、命名空间装配这些高级能力。
3. **端口不是一等公民。** 它有 link frame / joint frame，但没有“模块连接端口”这种更高层概念。
4. **更像描述机器人成品，不像描述模块库。** 你当前想做的是模块化机构编码，URDF 粒度偏系统级而不是模块原语级。

### 对本项目最值得借鉴的部分

- `link` / `joint` 作为基础运动学原语。
- `origin + axis + limit` 的最小描述集合。
- 明确区分几何、惯性、运动副、传感器等层次。

### 不建议照搬的部分

- 纯树结构假设。
- 把“连接关系”完全绑定在 parent-child link 上，而不引入显式端口。

### 对本项目的判断

URDF 更适合作为 **开环基础抽象参考**，不适合作为最终目标语言本体。

## 4.2 Xacro

### 定位

Xacro 不是独立机器人语义格式，而是 **URDF 的宏展开语言**。它的核心价值不是重新定义动力学对象，而是为 XML 提供：

- 宏
- 属性
- 参数化
- include
- 条件展开

本质上，Xacro 解决的是“URDF 太啰嗦、复用太差”的问题。

### 优点

1. **参数化能力强。** 很适合描述重复模块、镜像模块、左右对称支链。
2. **能把模块定义和实例化分开。** 这点对你的模块库非常重要。
3. **低门槛。** 它不改变 URDF 的下游生态，只是在上游增加生成层。
4. **适合做语言的前端层。** 你的 DSL 将来也可以采用“高层装配 -> 展开 -> 低层中间表示”的路线。

### 缺点

1. **它只负责展开，不负责语义验证。** 宏展开后是否形成合法机构，不是 Xacro 本身要解决的事。
2. **错误可能延迟暴露。** 文本层看起来正确，但展开结果的物理含义未必正确。
3. **过度宏化后可读性下降。** 宏系统很容易把模型写成“模板程序”，而不是“机构定义”。
4. **仍然受 URDF 底层限制。** 闭环、约束、图结构弱的问题并不会因为用了 Xacro 就消失。

### 对本项目最值得借鉴的部分

- 模块定义和模块实例化分层。
- 属性和表达式保留符号形式。
- include 与文件拆分。
- 条件化生成同一模块的不同变体。

### 不建议照搬的部分

- 只做文本预处理而不建立结构化中间语义。
- 让最终机构语义过度依赖宏展开副作用。

### 对本项目的判断

Xacro 最适合作为 **语言前端机制的参考**，而不是求解前的核心解释层。换句话说，它告诉你“模块库如何复用”，但不告诉你“闭环方程如何生成”。

---

## 4.3 SDFormat (SDF)

### 定位

SDF 是 Gazebo 生态的通用仿真描述格式，覆盖面比 URDF 更广。它不只描述机器人，还描述：

- world
- model
- link
- joint
- frame
- sensor
- light
- actor
- physics

相比 URDF，SDF 明显更像一个 **完整场景与物理系统描述语言**。

### 优点

1. **模型组合更成熟。** SDF 有更强的 nested model 和 model composition 设计。
2. **frame 与作用域更强。** 这一点非常值得你参考，因为你的模块连接本质就是 frame 连接。
3. **命名空间意识更强。** scoped names 直接解决嵌套模型中的引用问题。
4. **不只适合机器人树。** 它的组织方式比 URDF 更接近复杂机构和复杂仿真系统。

### 缺点

1. **格式较重。** 覆盖 world / physics / rendering / sensors 等大量概念，不适合作为纯运动学 DSL 直接照抄。
2. **通用性带来复杂性。** 对你阶段 A 来说，其中很多能力是无关噪声。
3. **闭环表达仍不等于自动约束生成。** 它能表达更多结构，但你的核心问题仍然是“如何翻译成符号方程”。

### 对本项目最值得借鉴的部分

- **作用域命名**：例如 `moduleA::submoduleB::frameC` 这种引用方式。
- **嵌套模型**：支持模块内部继续包含子模块。
- **显式 frame 体系**：不要只把连接点看成位置点，而要看成完整局部坐标系。
- **组合优先**：模型库定义与场景装配分开。

### 不建议照搬的部分

- world / scene / rendering / actor 这些与你当前运动学生成无关的层。
- 过度仿真导向的字段。

### 对本项目的判断

如果说 URDF 给你的是“骨架”，SDF 给你的就是“更成熟的组合与 frame 语义”。对你的 DSL 来说，SDF 的 **作用域、frame、嵌套模型** 非常值得吸收。

## 4.4 MJCF

### 定位

MJCF 是 MuJoCo 的模型描述语言。它不是单纯的机器人描述文件，而是一个 **编译式物理模型语言**。除了 bodies、joints、geoms 之外，它还内建：

- `frame`
- `include`
- `replicate`
- `attach`
- `default`
- `equality`
- `tendon`
- `actuator`
- `sensor`

而且很多 meta 元素会在编译后消失，只留下展开后的低层模型。这一点和你未来想做的“DSL -> 运动学 IR”非常接近。

### 优点

1. **编译期语义非常强。** `frame`、`replicate`、`attach` 等机制说明它不是简单存档格式，而是生成型语言。
2. **闭环/约束是显式对象。** `equality connect/weld/joint/tendon` 很值得参考。
3. **site / frame / tendon / actuator 都是一等公民。** 这比 URDF 的表达能力强很多。
4. **默认值与类机制成熟。** `default class` 能显著减少重复。
5. **局部 frame 思维极强。** 这与你的模块端口语义高度匹配。

### 缺点

1. **强绑定 MuJoCo 仿真语义。** 它的很多字段是为 MuJoCo solver 服务的，对你会是噪声。
2. **语言规模大。** 如果直接模仿整套 MJCF，会把你的项目拖入完整仿真 DSL，而不是聚焦运动学自动生成。
3. **对“机构模块库”并非原生最优抽象。** 它更偏 body tree + constraints + simulation objects，不是专门为可重构模块化机构设计。

### 对本项目最值得借鉴的部分

- **`frame` 元素**：纯坐标变换层，编译后可消失。
- **`attach` / `include` / `replicate`**：模块复用、拼装、重复阵列生成。
- **`equality`**：把闭环约束、耦合约束显式建模，而不是偷偷塞进求解器里。
- **`site`**：位置 / 姿态参考点独立于几何体。
- **meta-element 编译思想**：高层语言不必等于最终求解输入。

### 不建议照搬的部分

- 大量 MuJoCo 专属动力学、接触、渲染、执行器细节。
- 直接把你的 DSL 设计成“仿真配置文件”。

### 对本项目的判断

MJCF 是本次调查里对你 **最有启发性** 的参考之一。不是因为它最像你的目标，而是因为它已经证明：

- 高层装配语义
- 编译期展开
- 约束一等化
- 局部 frame 显式化

可以在同一门语言中共存。

## 4.5 Modelica.Mechanics.MultiBody

### 定位

Modelica MultiBody 不是 XML 格式，而是一个 **面向对象的多体系统建模语言/库体系**。它的核心不是“存一个机器人树”，而是“用组件和连接器描述机械系统”。

从官方说明可以直接看到它包含：

- World
- Joints
- Parts
- Forces
- Sensors
- Frames
- Interfaces

并且官方文档明确强调：它依赖符号变换和 DAE 处理能力，从而支持组件以面向对象方式连接成复杂机构，包含闭环。

### 优点

1. **闭环天然存在。** 它不是先假定树结构，再想办法绕过去。
2. **连接器语义强。** frame connector 是一等概念，这和你的模块端口非常接近。
3. **组件化成熟。** 结构件、关节、力元件、传感器都能统一组合。
4. **符号处理导向强。** 它的核心思想之一就是通过符号变换处理过定约束与闭环问题。

### 缺点

1. **过于重量级。** 它是一整套建模语言与求解生态，不只是一个轻量描述格式。
2. **门槛高。** 对阶段 A 来说，如果直接用 Modelica 风格重做一套，复杂度会过高。
3. **更偏动力学系统建模。** 你现在的焦点是运动学生成和解释器，不需要一开始就把 DAE 世界全搬进来。

### 对本项目最值得借鉴的部分

- **connector-first 思想**：模块连接语义应当比文件语法更核心。
- **组件 + 连接 + 约束** 三层分离。
- **闭环不是异常情况，而是正常图结构。**
- **中间表示应允许 overconstrained / constrained system 的显式存在。**

### 不建议照搬的部分

- 过度通用的方程系统建模能力。
- 一开始就设计完整 DAE 语言。

### 对本项目的判断

Modelica MultiBody 对你最有价值的不是语法，而是 **架构哲学**：不要把模块语言设计成“树形装配文件”，而要设计成“组件连接系统”。

## 4.6 Simscape Multibody

### 定位

Simscape Multibody 是图形化多体系统建模环境。对你当前项目来说，它不是外部参考，而是当前现实起点。你现在正是要从它迁移出去。

### 优点

1. **模块化连接非常直观。** block + frame port 的建模思路对模块机构特别友好。
2. **frame 概念明确。** 每个连接不是简单点连接，而是刚体 frame 间连接。
3. **闭环机构建模自然。** 这是它对你当前工作最关键的现实优势。
4. **求解器已集成。** 用户不必先自己写解释器和约束构造器。

### 缺点

1. **强环境依赖。** 你已经在路线文档中明确识别到这一点。
2. **模型语义对文本化不友好。** `slx` 可以拆解，但不适合直接当长期标准。
3. **模块定义隐含在图中。** 这正是你当前要解决的问题：把图模型提升为文本化、形式化模块库。
4. **不利于自动生成可移植符号方程。**

### 对本项目最值得借鉴的部分

- 模块连接口应当是 **frame / port**，不是裸坐标。
- 模块内部应允许由多个刚体、变换和关节组成，而不是只有一个 link。
- 图形模型可以被视作一种“高级可视前端”，但不应是底层权威表示。

### 不建议照搬的部分

- 图形化依赖。
- `slx` 作为长期主格式。

### 对本项目的判断

Simscape 最适合作为 **语义参考和验证基线**，不适合作为未来 DSL 载体。

## 4.7 Drake Model Directives / Drake Extensions

### 定位

Drake 同时支持 URDF、SDF、MJCF，并在其之上增加：

- YAML 的 Model Directives
- 对 URDF/SDF 的扩展 tag
- 组合与场景装配机制
- 额外约束与专用 joint 类型

它非常像一个现实中的“分层语言系统”：

- 底层格式继续用标准格式。
- 上层再用 directives 和 extensions 做装配与增强。

### 优点

1. **分层很清楚。** 基础模型描述与装配描述分开。
2. **组合能力强。** 支持把不同格式模型拼成复杂 scene。
3. **支持扩展约束。** 例如 ball constraint、tendon constraint、bushing、spring-damper。
4. **说明了扩展标准格式的现实路径。**

### 缺点

1. **依然偏工程集成层。** 它更关心把模型吃进 `MultibodyPlant`，不直接服务于你的符号生成目标。
2. **扩展机制较散。** 作为语言研究对象时，容易变成“标准格式 + 一堆专用补丁”。
3. **如果直接模仿，会让 DSL 变成某基础格式的寄生层。** 你当前更需要的是原生、统一的机构描述语义。

### 对本项目最值得借鉴的部分

- **装配层独立**：模块定义层和机构装配层分开。
- **作用域命名与跨模型引用**。
- **扩展标签只做增量能力，不破坏基础格式。**
- **约束与附加语义显式声明**，而不是靠解释器猜。

### 不建议照搬的部分

- 把核心语义完全寄生在别人格式之上。
- 用大量 ad-hoc 扩展标签替代统一 DSL 设计。

### 对本项目的判断

Drake 最值得借鉴的是 **语言分层策略**：

- 模块定义层
- 机构装配层
- 求解/约束配置层

这和你阶段 A 的解释器路线高度一致。

## 4.8 DH 参数体系

### 定位

DH 参数并不是完整机器人描述语言，而是一套针对串联机构的几何参数化方法。它擅长用少量参数表达关节间相对变换。

### 优点

1. **极度紧凑。** 对标准串联开环链非常高效。
2. **直接服务于运动学。** 这一点和你阶段 A 的目标贴近。
3. **容易手工验证。** 可作为解释器输出的交叉检验基线。

### 缺点

1. **表达范围窄。** 对模块化、分支、闭环、自定义坐标系适配都不自然。
2. **端口/模块概念缺失。**
3. **不适合描述机构库。** 它只适合描述某一条特定链。

### 对本项目最值得借鉴的部分

- 对简单开环支链，使用最小几何参数表示。
- 作为解释器前期测试样例的人工对照基线。

### 不建议照搬的部分

- 用 DH 作为整个 DSL 核心抽象。

### 对本项目的判断

DH 应作为 **局部验证工具或低级后端表达**，而不是上层模块语言。

|------|------|------|------|------|------|------|
| URDF | 机器人树描述 | 弱 | 中 | 弱 | `link/joint/origin/axis` 基本骨架 | 天然偏树，闭环弱 |
| Xacro | URDF 宏展开层 | 强 | 继承自 URDF | 弱 | 模板、参数、include、生成层 | 只做展开，不做语义 |
| SDF | 通用仿真模型格式 | 中到强 | 强 | 中 | 嵌套模型、scoped name、frame | 过重，超出阶段 A 需要 |
| MJCF | 编译式仿真模型语言 | 强 | 强 | 强 | `frame`、`attach`、`replicate`、`equality` | 强绑定 MuJoCo 语义 |
| Modelica MultiBody | 组件化多体建模体系 | 强 | 很强 | 很强 | connector-first、组件连接、闭环自然化 | 体系过重 |
| Simscape Multibody | 图形化多体建模环境 | 强 | 很强 | 很强 | frame port、模块图连接语义 | 文本化与可移植性弱 |
| Drake Directives / Extensions | 装配与扩展层 | 强 | 强 | 中到强 | 分层设计、扩展层、约束声明 | 更偏工程集成而非原生 DSL |
| DH 参数 | 串联机构几何参数法 | 无 | 弱 | 弱 | 简洁开环验证基线 | 不能当完整语言 |

## 6. 对本项目最重要的借鉴点

结合以上调查，如果目标是“为模块化闭环机构设计一套符号化语言，并写解释器生成 FK/IK 几何模型”，那么最值得吸收的设计不是某一门语言的整体，而是以下几条原则。

### 6.1 语言必须分层

至少分成三层：

1. **模块定义层**：定义模块类型、端口、局部 frame、内部关节与固定变换。
2. **机构装配层**：定义模块实例、实例参数、端口连接、末端参考系。
3. **求解配置层**：定义哪些变量是未知量、哪些量是任务输入、闭环误差选取哪些分量。

这一点综合借鉴自：

- Xacro 的生成层
- Drake directives 的装配层
- MJCF 的 meta-element 层

### 6.2 port / frame 必须是一等公民

不要把连接点设计成一个无方向的点。对模块化机构来说，连接对象必须至少是：

- 一个局部原点
- 一组局部轴方向
- 一个明确端口角色

这一点最强的参考来自：

- Simscape Multibody
- Modelica MultiBody
- SDF / MJCF 的 frame / site 思想

### 6.3 语言核心应是图，而不是树

即便第一版解释器先从开环树跑通，语言语义本身也不应被树结构绑死。模块化闭环机构本质上是一般图。

这一点是 URDF 最不该照搬的地方。

### 6.4 闭环约束必须显式建模

闭环不应该只是“连接后自动猜出来的副作用”。更合理的做法是：

- 语言允许声明闭环连接
- 解释器输出闭环误差候选项
- 求解配置再选择实际约束分量

这一点可借鉴：

- MJCF `equality`
- Drake constraint extensions
- Modelica 的约束连接思路

### 6.5 模块复用要通过参数化和实例化完成

不要靠复制粘贴模块定义。应支持：

- 模块类型 + 参数
- 模块实例 + 实例名
- 对称复制、重复阵列、变体配置

这一点可借鉴：

- Xacro 宏
- MJCF `replicate`
- SDF nested model

### 6.6 解释器必须生成稳定中间表示，而不是直接拼求解器输入

解释器不应一边解析文本一边直接生成 `subs` 或 `matlabFunction` 代码。更合理的路线是先生成一层运动学 IR，例如：

- `Body`
- `Frame`
- `Port`
- `Joint`
- `FixedTransform`
- `Constraint`
- `SymbolRegistry`

这一层再喂给现有 MATLAB 求解模板。这个思想最接近 MJCF 的编译式模型和 Modelica 的组件连接语义。

## 7. 对自定义语言的初步建议

如果完全面向当前项目需求，建议未来的自定义语言遵循下面这套组合策略：

### 建议保留的核心特性

- **URDF 的基础刚体/关节模型**
- **Xacro 的参数化与模板化思路**
- **SDF 的作用域命名与嵌套模型思路**
- **MJCF 的编译式 meta 元素和 equality 约束思路**
- **Modelica / Simscape 的 connector-first 思维**
- **Drake 的分层与扩展层策略**

### 建议避免的方向

- 做成只适合树结构的格式
- 过度依赖 XML 文本展开而没有语义 IR
- 直接把语言做成完整仿真配置系统
- 把闭环支持做成事后补丁

### 最适合你的语言轮廓

更合理的目标不是“另一个 URDF”，也不是“文本版 Simscape”，而是一门：

- **模块库优先**
- **端口 / frame 优先**
- **图结构优先**
- **约束显式化**
- **编译到符号运动学 IR**

的机构描述语言。

## 8. 对阶段 A 的直接启发

把这份调查映射回当前阶段 A，可以得到非常明确的工程建议：

1. **A.1 模块库定义**：主要借鉴 Simscape + SDF + MJCF 的 frame/port 思维。
2. **A.2 DSL v0**：主要借鉴 URDF 的基础 link/joint 结构，结合 Xacro/Drake 的分层装配思想。
3. **A.3 开环解释器**：先按 URDF 风格树遍历实现，但 IR 必须按一般图设计。
4. **A.4 闭环扩展**：重点借鉴 MJCF equality 和 Modelica 约束连接思路。
5. **A.5 接入求解器**：保持你在 `inverse-kinematics-solver-design` 里已经验证过的符号求解流程不变，只替换上游生成器。

换句话说，调查结果进一步支持你目前的路线判断：

- **关键不是先发明最复杂的语法。**
- **关键是先把模块定义和解释器抽象做对。**

## 9. 建议的下一步

基于本次调查，最自然的下一步不是继续扩展“调研范围”，而是开始收敛你自己的语言设计。我建议优先按这个顺序推进：

1. 先定义 **模块定义 schema v0**。
2. 再定义 **机构装配 DSL v0**。
3. 然后设计 **解释器内部 IR**。

只有这三层先稳住，后续 FK/IK 生成和闭环验证才会顺。
